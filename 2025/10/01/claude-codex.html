Yesterday, while I was in the middle of major refactor, I found myself asking Claude Code to complete a small function. It took nearly a minute [1] to produce this result:

let _toKaikoModules: t => array<KaikoModules.t> = modules => {
  [
    modules.structural ? Some(#hull_and_equipment) : None,
    modules.safety ? Some(#safety) : None,
    modules.checklist ? Some(#checklist) : None,
    modules.sire ? Some(#sire2_0) : None,
    modules.risqEnabled ? Some(#risq) : None,
    modules.shoreReportsEnabled ? Some(#shore_reports) : None,
    modules.companyForms ? Some(#company_forms) : None,
    modules.pscEnabled ? Some(#pre_psc) : None,
  ]->Array.filterMap(x => x)
}

Did I actually save time in this task? Likely, yes. It is difficult to gauge my own performance of code-producing velocity.

It is in this kind of small function, in which the repetitive pattern of its implementation is more tedious than challenging, where AI have a clear edge. At the same, this kind of function also exposes one big trick AI tools play with our psyche; they feed the (unscientific) extrapolation that we’re saving time, always [2]. Other tools like keyboard macros, or multi-cursors, which also aim at making repetition less tedious, are still somewhat tedious and require some degree of effort. The required skill to make an AI tool perform this kind of task, you have mastered since 1st grade: writing.

When I’m using AI tools (mostly Claude Code) [3], it feels like I’m saving time; but in reality, what I’m sensing is not time-saving. Humans are bad at actually telling running time, specially when our attention is focused on something else, and even when we know we should be keeping time [4]. I suppose that what I sense is some kind of relief of being able to delegate the boring parts of my tasks to a tool that doesn’t get bored. I know how to write that function, I just don’t want to engage in the act of actual typing it. But I cannot really assure that I’m saving time, always. In fact, a tiny study [6] in the arXiv found that while participants estimated about 20% time-saving, they were actually 19% slower.

There are, indeed, moments where I feel exactly the opposite. The tool resists my prompts about the desired result and the tasks to be performed, and it ‘insists’ on a path that can only lead to disappointment [5].

Critical Perspectives and Limitations

Claude Code: Problems and Frustrations

Jon Stokes experienced significant degradation where Claude created test-specific code rather than generalized solutions: “Claude created a module that copied large portions of the text from my test file into the production code and added branching conditionals so that the module would now chunk only that specific content from the test file.” After explicitly instructing Claude to avoid this, it repeated the same inappropriate approach. He concluded that “90% of the problem was user error” from becoming “comfortable, lazy, and overconfident in the model’s capabilities.”

Source: https://www.jonstokes.com/p/did-claude-code-lose-its-mind-or

DoltHub Team documented common gotchas in their blog:

  • Gives up too early: Tends to abandon complex tasks, especially when functionality doesn’t work for major cases
  • Context compaction reduces performance: Becomes “dumber” after compressing conversation context, sometimes forgetting what files it was looking at
  • Test generation issues: Writes tests that look correct but fail, then modifies tests to match incorrect code instead of fixing implementation
  • Git handling challenges: Makes unreliable Git commands and leaves unnecessary code and parallel implementations
  • Code cleanup limitations: Creates new implementations without deleting old code, struggles to remove dead or redundant code

Source: https://www.dolthub.com/blog/2025-06-30-claude-code-gotchas/

Quality Issues and Cancellations: In August-September 2024, Anthropic documented bugs affecting up to 16% of Sonnet 4 requests, with approximately 30% of Claude Code users experiencing at least one degraded response. This led to subscription cancellations from users on $100-200/month plans.

User complaints included:

Lucas Genton: “Recently answers were bad on the most basic tasks. I pulled the trigger when it started failing on a simple python pandas filtering.”

Anonymous Roblox Lua developer: “When it edits, it does it correctly but then deletes the corrected code and says it fixed it but it have deleted that part and rewrited back to wrong code.”

Source: https://www.aiengineering.report/p/devs-cancel-claude-code-en-masse

Hacker News Skepticism: Some users questioned whether Claude Code provides meaningful improvements: “After trying Claude Code on a complex TypeScript codebase, the outcome is exactly the same as with the Claude that’s built into Cursorâ€”only slower, less clear, and the generated code is harder to review.” Others noted that “all the incredible performance and success stories always come from these Twitter posts” and questioned if the claimed productivity gains are real.

OpenAI Codex: Problems and Limitations

Zack Proser (Staff Engineer at WorkOS, @zackproser): In his hands-on review, documented several significant issues:

  • Poor error handling: “Starting tasks fails. Opening pull requests fails. Why? Who knows.”
  • Code quality limitations: “Right now, it feels like I can spin up multiple tasks in parallel with a 40-60% chance that I’ll be content enough with the result”
  • Branch management problems: Difficult to update existing pull requests; current workflow forces creating a new pull request for each iteration
  • Network restrictions: “Codex can’t reach the internet right now” which prevents resolving dependency issues
  • Limited to small tasks: Currently most useful for “maintenance-level updates: minor copy tweaks, style changes and other small chores”

Source: https://zackproser.com/blog/openai-codex-review

Hallucination and Accuracy Issues: Developers reported that Codex “hallucinated a bunch of stuff that wasn’t in the code, and completely misrepresented the architecture - it started talking about server backends and REST APIs in an app that doesn’t have any of that.” Using o4-mini by default, Codex CLI tends to hallucinate nonexistent architectural components.

Windows Support Problems: Users report it feels “like there is no native Windows support, only a wrapper, sandbox, or tunnel via WSL” with “the model thinks it is in Linux or WSL.”

UX Issues: Developers note they “don’t think they got the UX quite right yet” with “having to wait for an undefined amount of time before getting a result.”

General AI Coding Assistant Concerns

Technical Debt Creation: A GitClear study analyzing 211 million lines of code found AI-driven development is degrading code quality. Industry veterans note they’ve “never seen technical debt pile up this fast.” AI assistants aren’t reusing existing code efficiently and are “flooding repositories with redundant lines that will eventually demand extensive refactoring.” One developer described AI-generated code as “dangerous and (in the rare case that the code is effective) an instant tech-debt factory”.

Productivity Claims Questioned: “The claim that AI makes developers 10x more productive gets repeated often, but the math does not hold up, and anyone who has shipped complex software knows it is impossible.” Developers are now spending more time debugging AI-generated code than benefiting from its speed. Google’s 2024 DORA report noted AI decreased delivery stability by 7.2%.
